# 第一部分 数据表示与传送

## 3.1 程序编码

### 3.1.1 从 C 到汇编再到机器代码

以一个简单的源文件为例：

```c title="mulstore.c"
long mult2(long, long);

void mulstore(long x, long y, long *dest) {
    *dest = mult2(x, y);
}
```

使用 `gcc` 编译时，几个常见参数的含义如下：

| 指令参数 | 说明 |
| --- | --- |
| `-Og` | 以便于调试和理解为目标进行优化，通常能较好保留程序结构 |
| `-O2` / `-O3` | 更积极的优化，生成更高效的代码，但代码结构可能变化较大 |
| `-S` | 生成汇编代码 |
| `-c` | 只编译为目标文件，不链接 |
| `-o` | 指定输出文件名 |

```bash
gcc -Og -o prog mulstore.c  # 直接编译并链接生成可执行文件
gcc -Og -S mulstore.c       # 生成汇编代码 mulstore.s
gcc -Og -c mulstore.c       # 生成目标二进制文件 mulstore.o
objdump -d mulstore.o       # 反汇编目标文件
```

编译器生成的汇编代码大致如下：

```asm title="mulstore.s"
mulstore:
    pushq   %rbx
    movq    %rdx, %rbx
    call    mult2
    movq    %rax, (%rbx)
    popq    %rbx
    ret
```

:::note
汇编文件中会包含很多以 `.` 开头的伪指令，例如 `.file`、`.text`、`.globl`。它们不是机器指令，而是给汇编器和链接器使用的说明信息。
本文悉数省略
:::

反汇编目标文件后，可以看到对应的机器码字节：

```asm
0000000000000000 <mulstore>:
    0:   53                   push   %rbx
    1:   48 89 d3             mov    %rdx,%rbx
    4:   e8 00 00 00 00       callq  ...
    9:   48 89 03             mov    %rax,(%rbx)
    c:   5b                   pop    %rbx
    d:   c3                   retq
```

### 3.1.2 调用约定和寄存器保存

`pushq %rbx` 和 `popq %rbx` 对应保存寄存器的值到栈上和恢复栈上的值到寄存器，这引出了函数调用约定中的两类保存策略：

- **调用者保存**（caller-saved）：函数调用前，调用者负责保存这些寄存器的值
- **被调用者保存**（callee-saved）：函数返回前，被调用者负责恢复这些寄存器的值

在 x86-64 System V ABI 中，常见规则如下：

- **调用者保存寄存器**：`%rax`, `%rcx`, `%rdx`, `%rsi`, `%rdi`, `%r8`, `%r9`, `%r10`, `%r11`
- **被调用者保存寄存器**：`%rbx`, `%rbp`, `%r12`, `%r13`, `%r14`, `%r15`
- **`%rsp`**：栈指针，由 ABI 单独约定，始终指向当前栈顶，不划入以上两类

函数参数按顺序优先通过寄存器传递：

| 参数位置 | 寄存器 |
| --- | --- |
| 第1个参数 | `%rdi` |
| 第2个参数 | `%rsi` |
| 第3个参数 | `%rdx` |
| 第4个参数 | `%rcx` |
| 第5个参数 | `%r8` |
| 第6个参数 | `%r9` |

因此在 `mulstore(long x, long y, long *dest)` 中：

- `x` 在 `%rdi`
- `y` 在 `%rsi`
- `dest` 在 `%rdx`

编译器把 `%rdx` 复制到 `%rbx`，是因为 `%rdx` 属于调用者保存寄存器，而后续的 `call mult2` 可能会破坏它；`%rbx` 是被调用者保存寄存器，可以安全保存 `dest`。

**返回值通常放在 `%rax` 中**。

:::note
关于寄存器名字的由来，可以看这篇文章：[探索常用寄存器名字的由来](https://www.cnblogs.com/nicere/p/17030075.html)
:::

### 3.1.3 指令后缀和数据大小

`movq` 中的后缀 `q` 表示 8 字节操作数。常见整数后缀如下：

| 后缀 | 字节数 | 含义 |
| --- | --- | --- |
| `b` | 1 | byte |
| `w` | 2 | word |
| `l` | 4 | long word / double word |
| `q` | 8 | quad word |

浮点指令常见后缀是 `ss` 和 `sd`，分别表示单精度和双精度标量浮点运算。

注意：在 x86-64 中，32 位结果写入通用寄存器时会自动**零扩展**到 64 位；而对 64 位寄存器使用 32 位立即数时，立即数会先做**符号扩展**。

## 3.2 数据传送与寻址

### 3.2.1 `mov` 指令

`mov` 是最常见的数据传送指令，语义是把源操作数**复制**到目的操作数。

源操作数可以是：

- 立即数
- 寄存器
- 内存引用

目的操作数可以是：

- 寄存器
- 内存引用

但不能同时都是内存引用，也就是说，**不能直接做内存到内存的复制**。

一些常见约束：

- 复制立即数到 64 位寄存器时，通常使用 32 位立即数（截取）并进行符号扩展
- 若需要真正的 64 位立即数，应使用 `movabs`
- `movl` 写入 64 位寄存器时，会把高 32 位**清零**（而非符号扩展）
- `movz*` 和 `movs*` 用于零扩展和符号扩展

常见扩展指令如下：

| 指令 | 含义 |
| --- | --- |
| `movzbw` | byte -> word，零扩展 |
| `movzbl` | byte -> long，零扩展 |
| `movzbq` | byte -> quad，零扩展 |
| `movzwl` | word -> long，零扩展 |
| `movzwq` | word -> quad，零扩展 |
| `movsbw` | byte -> word，符号扩展 |
| `movsbl` | byte -> long，符号扩展 |
| `movsbq` | byte -> quad，符号扩展 |
| `movswl` | word -> long，符号扩展 |
| `movswq` | word -> quad，符号扩展 |
| `movslq` | long -> quad，符号扩展 |

`cltq` 的作用是把 `%eax` 符号扩展到 `%rax`，等价于 `movslq %eax, %rax`。

### 3.2.2 地址形成方式

内存引用常写成`Imm(r_b, r_i, s)` 的形式，对应的地址计算公式为：

$$
Imm(r_b, r_i, s) = Imm + R[r_b] + s \cdot R[r_i]
$$

其中：

- `Imm` 是偏移量
- `r_b` 是基址寄存器
- `r_i` 是变址寄存器
- `s` 是比例因子，只能取 `1, 2, 4, 8`

常见寻址形式如下：

| 格式 | 含义 |
| --- | --- |
| `$Imm` | 立即数寻址（立即数本身） |
| `r_a` | 寄存器寻址 |
| `Imm` | 绝对寻址（立即数作为地址） |
| `(r_a)` | 间接寻址 |
| `Imm(r_b)` | 基址 + 偏移 |
| `(r_b, r_i)` | 基址 + 变址 |
| `Imm(r_b, r_i)` | 基址 + 偏移 + 变址 |
| `(, r_i, s)` | 比例变址 |
| `Imm(, r_i, s)` | 偏移 + 比例变址 |
| `(r_b, r_i, s)` | 基址 + 比例变址 |
| `Imm(r_b, r_i, s)` | 偏移 + 基址 + 比例变址 |

### 3.2.3 栈上的传送

`pushq %rax` 可以理解为：

```asm
subq $8, %rsp
movq %rax, (%rsp)
```

`popq %rax` 可以理解为：

```asm
movq (%rsp), %rax
addq $8, %rsp
```

栈是后进先出结构，`push` 会让栈顶向低地址移动，`pop` 会让栈顶向高地址移动。栈中并不会清除旧数据，只是 `rsp` 指向的位置发生变化。

# 第二部分 运算与控制流

## 3.3 算术与逻辑运算

### 3.3.1 `leaq`

`leaq` 看起来像"load effective address"，但它的真正用途是计算地址表达式并把结果写入寄存器，它不访问内存。

例如：

```asm
leaq 7(%rdx, %rdx, 4), %rax
```

计算的是：

$$
7 + R[rdx] + 4 \cdot R[rdx]
$$

因此 `leaq` 常常被编译器拿来做整数乘法、加法和合并计算。

例如：

```c
long scale(long x, long y, long z) {
    return x + 4 * y + 12 * z;
}
```

可以写成：

```asm
scale:
    leaq (%rdi, %rsi, 4), %rax   # rax = x + 4*y
    leaq (%rdx, %rdx, 2), %rdx   # rdx = z + 2*z = 3*z
    leaq (%rax, %rdx, 4), %rax   # rax = (x + 4*y) + 4*(3*z) = x + 4*y + 12*z
    ret
```

因为 `leaq` 的比例因子只能是 `1, 2, 4, 8`，不能直接写 `12`，所以编译器将 `12*z` 拆成两步：先用 `leaq` 计算 `3*z`，再乘以 4。

### 3.3.2 一元和二元运算

常见的一元运算：

| 指令 | 描述 |
| --- | --- |
| `inc D` | `D = D + 1` |
| `dec D` | `D = D - 1` |
| `neg D` | `D = -D` |
| `not D` | `D = ~D` |

常见的二元运算：

| 指令 | 描述 |
| --- | --- |
| `add S, D` | `D = D + S` |
| `sub S, D` | `D = D - S` |
| `imul S, D` | `D = D * S` |
| `xor S, D` | `D = D ^ S` |
| `or S, D` | `D = D \| S` |
| `and S, D` | `D = D & S` |

这里的 `imul S, D` 是双操作数形式，结果**截断保留低位**，直接写回 `D`。与之不同，一元形式 `imulq S` 会把 `%rax` 与 `S` 相乘，得到 128 位结果并分别存入 `%rdx`（高64位）和 `%rax`（低64位），通常用于需要完整大数结果的场景。

这些指令往往会同时设置条件码。

**整数运算溢出的行为：** 在 C 语言层面，**有符号整数**的溢出属于**未定义行为**，编译器可能做各种激进优化；**无符号整数**溢出则由标准明确规定为模 $2^w$ 回绕。理解这一点对调试整数相关的 bug 非常重要。

### 3.3.3 移位运算

| 指令 | 描述 |
| --- | --- |
| `sal k, D` | shift arithmetic left，算术左移 |
| `shl k, D` | shift logical left，逻辑左移 |
| `sar k, D` | shift arithmetic right，算术右移 |
| `shr k, D` | shift logical right，逻辑右移 |

移位量 `k` 可以是立即数，也可以来自寄存器 `%cl`。无论哪种来源，硬件总是只取移位量的低 $m$ 位，$m=\log_2(w)$：

- 32 位操作数取低 5 位（相当于模 $2^5$）
- 64 位操作数取低 6 位（相当于模 $2^6$）

对立即数而言，汇编器通常只接受合法范围（$0$ 到 $w-1$）内的值，因此截断在实践中几乎不可见，但硬件行为始终存在。

### 3.3.4 整数除法指令

有符号和无符号除法都需要将被除数扩展为操作数位宽的两倍，因此除法指令通常与扩展指令配对使用。

关键扩展指令：

| 指令 | 作用 |
| --- | --- |
| `cqto` | Convert Quadword to Octoword，将 `%rax` 符号扩展到 `%rdx:%rax`（128 位），为有符号除法准备 |

除法指令：

| 指令 | 运算 | 结果 |
| --- | --- | --- |
| `idivq S` | 有符号除法 `%rdx:%rax ÷ S` | 商 → `%rax`，余数 → `%rdx` |
| `divq S` | 无符号除法 `%rdx:%rax ÷ S` | 商 → `%rax`，余数 → `%rdx` |

例如，计算两个 `long` 相除：

```c
long idiv(long x, long y) {
    return x / y;
}
```

对应的典型汇编：

```asm
idiv:
    movq  %rdi, %rax     # 被除数 x 放入 rax
    cqto                 # 符号扩展到 rdx:rax
    idivq %rsi           # 除以 y，商在 rax
    ret
```

### 3.3.5 条件码

算术和逻辑运算会更新条件码，常见条件码如下：

| 条件码 | 含义 |
| --- | --- |
| `CF` | carry flag，进位/借位标志，无符号运算相关 |
| `ZF` | zero flag，结果是否为 0 |
| `SF` | sign flag，结果符号位 |
| `OF` | overflow flag，有符号溢出标志 |

`cmp S, D` 实际上计算的是 `D - S`，但不保存结果，只更新条件码。`test S, D` 计算的是 `D & S`，同样只更新条件码。

### 3.3.6 条件跳转与跳转指令编码

#### `SETcc` 指令

`SETcc` SET if Condition Code，一系列条件设置指令，根据条件码的状态把 0 或 1 写入目的寄存器的最低字节（8位）。 常见的有符号比较：

| 指令 | 条件（写入值） | 描述 |
| --- | --- | --- |
| `sete` | `ZF` | set if equal |
| `setne` | `!ZF` | set if not equal |
| `setl` | `SF ^ OF` | set if less (signed) |
| `setle` | `(SF ^ OF) \| ZF` | set if less or equal (signed) |
| `setg` | `!(SF ^ OF) & !ZF` | set if greater (signed) |
| `setge` | `!(SF ^ OF)` | set if greater or equal (signed) |

无符号比较常用：

| 指令 | 条件（写入值） | 描述 |
| --- | --- | --- |
| `seta` | `!CF & !ZF` | set if above (unsigned) |
| `setae` | `!CF` | set if above or equal (unsigned) |
| `setb` | `CF` | set if below (unsigned) |
| `setbe` | `CF \| ZF` | set if below or equal (unsigned) |

#### 条件跳转指令

常见条件跳转：

| 指令 | 条件 | 描述 |
| --- | --- | --- |
| `jg` | `!(SF ^ OF) & !ZF` | jump if greater (signed) |
| `jge` | `!(SF ^ OF)` | jump if greater or equal (signed) |
| `jl` | `SF ^ OF` | jump if less (signed) |
| `jle` | `(SF ^ OF) \| ZF` | jump if less or equal (signed) |

#### 跳转指令的编码

条件跳转和 `jmp` 无条件跳转的目标地址通常以 **PC 相对偏移**（PC-relative）形式编码。指令中存储的是目标地址相对于下一条指令地址的**偏移量**：

$$
\text{目标地址} = \text{下一条指令地址} + \text{偏移量}
$$
这种编码方式使得代码具有**位置无关性**（position-independence），即代码可以被加载到内存的任意位置而无需修改跳转指令，因为跳转目标之间的相对距离保持不变。

#### `switch` 语句与跳转表

`switch` 语句在分支较多且 case 值比较密集时，常会被编译为跳转表。跳转表存放在 `.rodata` 段，本质上是一个函数地址数组，每个条目占 8 字节。编译器根据 case 值作为索引，通过一条间接跳转指令直接跳到对应分支：

```c
void switch_eg(long x, long n, long *dest) { 
    long val= x; 
    switch (n) { 
        case 100: val*= 13; break; 
        case 102: val+= 10; 
        case 103: val += 11; break; 
        case 104: 
        case 106: val*= val; break; 
        default: val= 0; 
    } 
    *dest = val; 
}
```
```asm
switch_eg:
    subq  $100, %rsi            # n - 100
    cmpq  $6, %rsi              # 比较 n - 100 和 6
    ja  .L8                     # 如果 n - 100 > 6，跳转到 default 分支
    jmp  *.L4(,%rsi,8)          # 跳转表，n - 100 作为索引
    .section  .rodata
    .align 8                    # 跳转表，8 字节对齐
.L4:
    .quad  .L7                  # Case 100
    .quad  .L8                  # Case 101 (不存在，跳转到 default)
    .quad  .L6                  # Case 102
    .quad  .L5                  # Case 103
    .quad  .L3                  # Case 104
    .quad  .L8                  # Case 105 (不存在，跳转到 default)
    .quad  .L3                  # Case 106
.L7:
    leaq  (%rdi,%rdi,2), %rax 
    leaq  (%rdi,%rax,4), %rdi   # val = x + 4 * (x + 2*x) = 13*x
    jmp  .L2
.L6:
    addq  $10, %rdi             # val = x + 10
.L5:
    addq  $11, %rdi             # val = x + 11
.L2:
    movq  %rdi, (%rdx)          # *dest = val
    ret                         # 返回
.L3:
    imulq  %rdi, %rdi           # val = x * x
    jmp  .L2
.L8:
    movl  $0, %edi              # val = 0
    jmp  .L2
```
这样一次跳转就能完成 N 路分支，而无需逐一比较。
当 case 值稀疏时，编译器通常退化为链式比较或二分查找，而不生成跳转表。

### 3.3.7 条件传送

条件传送指令可以根据条件码的状态选择性地把一个寄存器的值复制到另一个寄存器，而不需要跳转。这对于减少分支预测失败带来的性能损失非常有用。

| 指令 | 条件 |
| --- | --- |
| `cmove S, R` | `ZF` |
| `cmovne S, R` | `~ZF` |
| `cmovs S, R` | `SF` |
| `cmovns S, R` | `~SF` |
| `cmovg S, R` | `~(SF ^ OF) & ~ZF` |
| `cmovge S, R` | `~(SF ^ OF)` |
| `cmovl S, R` | `SF ^ OF` |
| `cmovle S, R` | `(SF ^ OF) \| ZF` |
| `cmova S, R` | `~CF & ~ZF` |
| `cmovae S, R` | `~CF` |
| `cmovb S, R` | `CF` |
| `cmovbe S, R` | `CF \| ZF` |

例如：

```c
long cmovdiff(long x, long y) {
    long rval = x - y;
    long eval = y - x;
    if (x < y)
        rval = eval;
    return rval;
}
```
```asm
cmovdiff:
    movq %rdi, %rdx     
    subq %rsi, %rdx     # rdx = x - y
    movq %rsi, %rax     
    subq %rdi, %rax     # rax = y - x
    cmpq %rsi, %rdi     # 设置条件码，比较 x 和 y
    cmovge %rdx, %rax   # 如果 x >= y，则 rax = rdx (x - y)，否则保持 rax = (y - x)
    ret
```

不过条件传送并非总是更优。当两个分支中任意一侧存在副作用（如指针解引用、函数调用），或者某一侧的计算本身开销很大时，编译器不会使用 `cmov`——因为两侧都会被求值，副作用无法回避，开销也无法节省。条件传送只适合两侧都是廉价无副作用的纯计算。在较高优化级别（如 `-O2`）下，编译器会更积极地生成 `cmov`，但仍受上述前提约束。

### 3.3.8 循环的机器级实现

C 语言中的三种循环结构——`do-while`、`while` 和 `for`——都会被翻译为基于条件码和条件跳转的组合。理解它们各自的翻译模式，是读懂控制流代码的关键。

#### `do-while` 循环

`do-while` 是最基础的循环形式：循环体至少执行一次，条件判断在循环体之后。汇编结构非常直接，就是“循环体 + 向后跳转”：

```c
do
    body;
while (test);
```

对应汇编模式：

```asm
loop:
    <body 的指令>
    <计算 test，设置条件码>
    jXX  loop          # 条件成立则跳回循环开始
```

#### `while` 循环的两种翻译策略

`while` 循环的翻译有两种常见方式：

**跳转到中间**（jump-to-middle）：先无条件跳转到尾部的条件测试，再从测试跳回循环体。这是 `-Og` 下常见的模式，代码结构更直接体现原始逻辑。

```c
while (test)
    body;
```

```asm
    jmp test
loop:
    <body 的指令>
test:
    <计算 test，设置条件码>
    jXX  loop         # 条件成立则跳回
```

**保护型 do-while**（guarded-do）：在较高优化级别（如 `-O1`）下常用。先在循环前加一次条件判断，若初始条件就不成立则直接跳过；循环体内部退化为 `do-while` 形式，省去了一次额外的无条件跳转。

```asm
    <计算 test，设置条件码>
    jXX  done         # 初次条件不满足则跳过循环
loop:
    <body 的指令>
    <计算 test，设置条件码>
    jXX  loop
done:
```

#### `for` 循环

`for` 循环的翻译本质上等价于将初始化、条件、步进三部分展开后，套用 `while` 的模式：

```c
for (init; test; update)
    body;
```

等价于：

```c
init;
while (test) {
    body;
    update;
}
```

因此它所用的汇编结构与 `while` 相同，只是循环体末尾额外包含了步进代码。

# 第三部分 过程调用

## 3.4 过程调用与栈

### 3.4.1 栈帧

函数调用时，系统会在栈上为函数分配一块连续内存以保存必要信息，这一片区域称为**栈帧**（stack frame）。一个典型函数调用会保存：

- 返回地址
- 调用者的基指针（如`%rbp`）
- 寄存器保存区（保存需要恢复的寄存器状态）
- 参数传递区（当参数数量超过寄存器可用时）
- 局部变量
- 临时计算结果

栈增长方向是向低地址扩展的，因此分配栈帧通常通过 `subq` 指令调整 `%rsp` 向下移动来完成。函数返回时，栈帧被销毁，`%rsp` 恢复到调用前的位置。

`call` 指令会把返回地址压入栈中，并把控制权转移给被调用函数。`ret` 指令会从栈顶弹出返回地址并跳回调用点。

### 3.4.2 栈对齐

System V AMD64 ABI 要求在执行 `call` 指令之前，`%rsp` 必须对齐到 16 字节边界（`movaps` 指令要求）。`call` 执行后会把 8 字节返回地址压栈，因此函数入口处 `%rsp` 是 16 的倍数减 8，即对 8 字节对齐。函数序言中常见的 `subq $N, %rsp` 里，N 的值正是为了在分配局部变量空间的同时，把 `%rsp` 重新恢复到 16 字节对齐。

### 3.4.3 参数传递

在 System V ABI 下：

- 前 6 个整数/指针参数通过 `%rdi`, `%rsi`, `%rdx`, `%rcx`, `%r8`, `%r9` 传递
- 前 8 个浮点参数通过 `%xmm0` 到 `%xmm7` 传递
- 超过寄存器数量的参数通过栈传递

通过栈传递的参数也遵循对齐规则，通常按 8 字节槽位处理。

### 3.4.4 一个局部变量示例

```c
long caller() {
    long arg1 = 534;
    long arg2 = 1057;
    long sum = swap_add(&arg1, &arg2);
    long diff = arg1 - arg2;
    return sum * diff;
}
```

其核心逻辑是：局部变量 `arg1` 和 `arg2` 被放在栈上，函数通过它们的地址调用 `swap`，随后再读取被修改后的值继续计算。

```asm
caller:
    subq $16, %rsp          # 为 arg1 和 arg2 分配空间
    movq $534, (%rsp)       # arg1 = 534
    movq $1057, 8(%rsp)     # arg2 = 1057
    leaq 8(%rsp), %rsi      # 第二个参数地址
    movq %rsp, %rdi         # 第一个参数地址
    call swap_add           # 调用 swap_add(&arg1, &arg2)
    movq %rax, (%rsp)       # sum = swap_add 的返回值
    subq 8(%rsp), %rax      # diff = arg1 - arg2
    imulq (%rsp), %rax      # sum * diff
    addq $16, %rsp          # 恢复栈指针
    ret
```

### 3.4.5 递归与被调用者保存寄存器

```c
long rfact(long n) {
    if (n <= 1)
        return 1;
    return n * rfact(n - 1);
}
```

递归调用中，当前参数必须在再次调用前保存下来，因此常使用被调用者保存寄存器，例如 `%rbx`。

```asm
rfact:
    cmpq $1, %rdi
    jle .L1
    pushq %rbx
    movq %rdi, %rbx
    leaq -1(%rdi), %rdi
    call rfact
    imulq %rbx, %rax
    popq %rbx
    ret
.L1:
    movl $1, %eax
    ret
```

### 3.4.6 定长数组

二维数组在内存中按**行优先**（row-major）存放。

```c
#define N 16
typedef int fix_matrix[N][N];

int matrix(fix_matrix A, fix_matrix B, long i, long k) {
    long j;
    int result = 0;
    for (j = 0; j < N; j++)
        result += A[i][j] * B[j][k];
    return result;
}
```

地址计算可写成：

$$
A[i][j] = A_{base} + (i \cdot N + j) \cdot 4
$$

$$
B[j][k] = B_{base} + (j \cdot N + k) \cdot 4
$$

编译器通常会对这类循环做**强度削弱**（strength reduction）：不在每次迭代中重新计算完整的乘法，而是维护一个步进指针。以 `B` 为例，每次迭代 `j` 增加 1，对应地址步进 `N * sizeof(int) = 64` 字节，因此可以用累加指针代替乘法：

```asm
matrix:
    salq $6, %rdx
    addq %rdx, %rdi             # A[i][0] = A + i * 64
    leaq (%rsi,%rcx,4), %rax    
    leaq 1024(%rax), %rsi       # B[15][k] = B + k * 4 + 1024
    movl $0, %ecx               # result = 0
.L2:
    movl (%rdi), %edx
    imull (%rax), %edx          
    addl %edx, %ecx             # result += A[i][j] * B[j][k]
    addq $4, %rdi               # A[i][j] -> A[i][j+1]
    addq $64, %rax              # B[j][k] -> B[j+1][k]
    cmpq %rsi, %rax             # 检查是否到达 B[15][k] 的地址
    jne .L2
    movl %ecx, %eax
    ret
```

### 3.4.7 变长栈帧

当局部变量大小在编译期无法确定时，就会出现变长栈帧，例如 `alloca` 或可变长数组相关的代码。

```c
long vframe(long n, long idx, long *q) {
    long i;
    long *p[n];
    p[0] = &i;
    for (i = 1; i < n; i++)
        p[i] = q;
    return *p[idx];
}
```

这类代码通常需要使用 `%rbp` 作为帧指针（frame pointer）来记录栈顶的初始位置，因为 `%rsp` 会随着动态分配不断变化，而固定偏移的局部变量仍需要稳定的引用基准。一般步骤是：
1. 函数序言中保存调用者的 `%rbp`，并把 `%rbp` 设置为当前 `%rsp` 的值
2. 调整并对齐 `%rsp` 来分配局部变量空间
3. 函数结尾恢复 `%rbp` 和 `%rsp`

# 第四部分 数据结构与内存安全

## 3.5 结构体、联合体与对齐

### 3.5.1 对齐规则

x86-64 System V ABI 要求 **K 字节大小的基本对象，其地址必须是 K 的倍数**。x86 架构本身允许非对齐访问，但会带来性能惩罚；编译器遵守 ABI 对齐约定，是为了保证效率和与其他代码的互操作性。

结构体中的成员为了满足对齐要求，编译器会在成员之间插入填充字节（padding），结构体末尾也可能需要填充，使数组中的每个元素都满足对齐。

### 3.5.2 结构体示例

```c
struct S1 {
    int i;
    char c;
    int j;
};
```

`S1` 的布局通常如下：

- `i` 占 4 字节
- `c` 占 1 字节
- `c` 和 `j` 之间需要插入 3 字节填充
- 总大小通常是 12 字节

```c
struct S2 {
    int i;
    int j;
    char c;
};
```

`S2` 的成员加起来虽然只有 9 字节，但为了保证结构体**数组**中每个元素都满足对齐要求，末尾通常还要补 3 字节，使总大小变成 12 字节。

### 3.5.3 联合体

联合体（union）中的所有成员共享同一块内存，因此联合体大小等于其最大成员的大小。

联合体适合表示互斥的多种解释方式，例如：

```c
unsigned long double2bits(double d) {
    union {
        double d;
        unsigned long l;
    } temp;

    temp.d = d;
    return temp.l;
}
```

工程上常见的做法是给联合体配一个枚举类型的标签（type），明确当前存放的是哪一种数据。

## 3.6 缓冲区溢出

缓冲区溢出发生在程序向缓冲区写入数据时越过边界，覆盖了相邻内存。它可能造成：

- 程序崩溃
- 数据被篡改
- 返回地址被覆盖
- 进一步的代码执行风险

经典危险函数包括 `gets`。它不检查缓冲区大小，现代 C 标准中也已经不再推荐使用，实际应改用 `fgets` 等安全接口。

### 3.6.1 常见防护

- **栈随机化**：通过地址空间布局随机化（ASLR），使得栈、堆和库的地址在每次程序运行时都不同，增加攻击者猜测正确地址的难度。

- **栈破坏检测**：在缓冲区和控制信息之间加入 canary（金丝雀值），函数返回前检查其是否被破坏。canary 的典型逻辑是：函数进入时保存一个随机值，返回前比较是否一致；若不一致，则说明发生了越界写。

- **限制可执行代码区域**：通过 NX 位（No-eXecute）或类似机制，使数据区和栈不可执行，阻止将 shellcode 直接注入栈上执行。

NX 并非终点。攻击者可以利用**返回导向编程**（ROP，Return-Oriented Programming）绕过 NX：不注入新代码，而是把已有代码中以 `ret` 结尾的短片段（gadget）串联起来，通过覆盖栈上的返回地址链条依次执行这些片段，最终完成任意操作。因此现代系统通常需要将 ASLR、栈保护和 NX 联合使用。

# 第五部分 浮点代码

## 3.7 浮点代码

### 3.7.1 浮点体系结构

早期 x86 使用 x87 浮点单元，后来逐步过渡到 SSE、SSE2 和 AVX。现代 x86-64 浮点运算主要基于 XMM/YMM 寄存器：

- `%xmm0` 到 `%xmm15`：128 位寄存器
- `%ymm0` 到 `%ymm15`：256 位寄存器，低 128 位与对应的 XMM 寄存器重叠

标量浮点运算通常只使用寄存器低位的一部分：

- `float` 使用低 32 位
- `double` 使用低 64 位

XMM 寄存器按 System V ABI 属于**调用者保存**，被调用函数可以自由修改它们，调用者若需保留其中的值，必须在调用前自行保存。

### 3.7.2 浮点参数传递

浮点参数和返回值通过 XMM 寄存器传递：

- 第 1 到第 8 个浮点参数使用 `%xmm0` 到 `%xmm7`
- 返回值通常放在 `%xmm0`

混合参数时，整数和指针仍走通用寄存器，浮点数走 XMM 寄存器。

### 3.7.3 常见浮点指令

| 单精度 | 双精度 | 含义 |
| --- | --- | --- |
| `vaddss` | `vaddsd` | Vector ADD Scalar Single-precision / Double-precision floating-point |
| `vsubss` | `vsubsd` | Vector SUB Scalar Single-precision / Double-precision floating-point |
| `vmulss` | `vmulsd` | Vector MUL Scalar Single-precision / Double-precision floating-point |
| `vdivss` | `vdivsd` | Vector DIV Scalar Single-precision / Double-precision floating-point |
| `vmaxss` | `vmaxsd` | Vector MAX Scalar Single-precision / Double-precision floating-point |
| `vminss` | `vminsd` | Vector MIN Scalar Single-precision / Double-precision floating-point |
| `vsqrtss` | `vsqrtsd` | Vector SQRT Scalar Single-precision / Double-precision floating-point |

和整数运算操作不同， AVX 浮点操作不能以立即数值作为操作数。相反，编译器必须为所有的常量值分配和初始化存储空间。然后代码再把这些值从内存读入。

### 3.7.4 转换、比较和位级操作

常见浮点转换指令包括：

| 指令 | 含义 |
| --- | --- |
| `vcvtsi2ss` | Vector ConVerT Signed Integer to Scalar Single-precision floating-point |
| `vcvtsi2sd` | Vector ConVerT Signed Integer to Scalar Double-precision floating-point |
| `vcvttss2si` | Vector ConVerT with Truncate Scalar Single-precision TO Signed Integer |
| `vcvttsd2si` | Vector ConVerT with Truncate Scalar Double-precision TO Signed Integer |

注意：浮点转整数时，若浮点值超出目标整数类型的可表示范围（例如 `(int)1e10`），C 语义是**未定义行为**，不能假设结果是简单截断。

浮点比较使用：

| 指令 | 含义 |
| --- | --- |
| `vucomiss` | Vector Unordered COMpare Scalar Single-precision floating-point |
| `vucomisd` | Vector Unordered COMpare Scalar Double-precision floating-point |

浮点比较结果通过 CF、ZF、PF（Parity Flag，奇偶标志符） 反映。
- PF 对于整数操作，当最近的一次算术或逻辑运算产生的值的最低位字节是偶校验的（即这个字节中有偶数个 1 ），那么就会设置这个标志位。
- 不过对于浮点比较，当两个操作数中任一个是 `NaN` 时，会设置该位。

与整数比较使用 SF/OF 不同。因此浮点比较后**必须配合无符号条件跳转指令**（`ja`、`jae`、`jb`、`jbe`、`je`），而非有符号跳转（`jg`、`jl`）。当某一操作数为 NaN 时，比较结果为"无序"，此时 `PF=1`，可通过 `jp` 检测。

位级操作也可用于浮点代码中，例如：

| 指令 | 含义 |
| --- | --- |
| `vxorps` / `vxorpd` | 按位异或操作 |
| `vandps` / `vandpd` | 按位与操作 |
