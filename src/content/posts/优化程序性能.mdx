# 第一部分 基础概念

## 5.1 性能指标

### 5.1.1 时间复杂度与实际运行时间

程序的运行时间受多个因素影响：

- **算法复杂度**：$O(n)$ 或 $O(n^2)$ 等
- **数据规模**：输入大小 $n$
- **处理器特性**：缓存、流水线、功能单元
- **编译器优化**：代码重排、指令融合等

对于有大量数据并行处理的循环程序，常见的时间模型是：

$$T(n) = a + b \cdot n$$

其中 $a$ 是**开销常数**（设置、初始化等），$b$ 是**线性因子**（每个元素的处理时间）。

### 5.1.2 CPE（Cycles Per Element）

**CPE** 是衡量**每个数据元素**处理时间的指标，定义为：

$$
\text{CPE} = \frac{\text{总时钟周期数} - \text{开销周期数}}{\text{处理的元素数}}
$$

例如，处理 1000 个浮点数，总耗时 9000 周期，其中开销 500 周期，则：

$$
\text{CPE} = \frac{9000 - 500}{1000} = 8.5
$$

这表示处理每个元素平均需要 8.5 个时钟周期。

:::note
在实际测量中，开销周期数可能难以准确分离，因此通常直接使用总周期数除以元素数来估算 CPE，在总周期数远大于开销周期数的情况下，这种近似是合理的。
:::

**CPE 的作用**：

1. 独立于数据规模，方便对比不同实现的性能
2. 反映处理器如何处理数据的实际能力
3. 指导我们找到性能瓶颈

## 5.2 优化的三个层次

一个完整的优化策略包含三个层次：

| 层次           | 手段                       | 限制条件           |
| -------------- | -------------------------- | ------------------ |
| **算法层次**   | 选择更优的数据结构和算法   | 受问题本质限制     |
| **程序层次**   | 代码重组、减少不必要操作   | 必须保持程序正确性 |
| **处理器层次** | 利用缓存、流水线、并行执行 | 取决于处理器能力   |

本章主要关注程序层次和处理器层次的优化。

# 第二部分 编译器的能力与局限

编译器可以进行很多优化，但也有重要的局限。理解这些限制有助于我们知道何时需要手动优化。

## 5.3 内存别名引用（Memory Aliasing）

**问题**：编译器无法确定两个指针是否指向同一地址。

例如：

```c
void add1(long *xp, long *yp){
    *xp += *yp;   // 读 yp，写 xp
    *xp += *yp;   // 读 yp，写 xp
}

void add2(long *xp, long *yp){
    *xp += 2 * (*yp);  // 只读 yp 一次
}
```

看起来 `add2` 更高效（3 次内存引用 vs 6 次）。但若 `xp` 和 `yp` 指向同一地址，设该地址初始值为 $y$：

- `add1`：第一次 `*xp += *yp` 后，`*xp` 变为 $y + y = 2y$；第二次执行时 `*xp` 和 `*yp` 都已是 $2y$，相加后变为 $2y + 2y = 4y$。最终结果是 $4y$。
- `add2`：`*xp += 2*(*yp)` 直接计算 $y + 2y = 3y$。最终结果是 $3y$。

**两个函数的语义不同**（$4y$ 与 $3y$）。因此编译器必须假设指针可能重叠，无法把 `add1` 优化成 `add2` 的形式。

## 5.4 函数调用的副作用

编译器无法推断函数是否有副作用。

```c
long f();

long func1(){
    return f() + f() + f() + f();  // 4 次调用
}

long func2(){
    return 4 * f();  // 1 次调用
}
```

如果 `f()` 内部有状态：

```c
static long counter = 0;
long f(){
    return counter++;  // 每次调用返回不同值
}
```

则 `func1` 依次拿到 `0, 1, 2, 3` 四次返回值，结果为 $0+1+2+3=6$；而 `func2` 只调用一次 `f()`，拿到 `counter` 当前值 `0`，结果为 $4 \times 0 = 0$。

编译器不能假设 `f()` 没有副作用，所以不能将 4 次调用优化为 1 次。

**结论**：某些优化编译器无法自动执行，需要程序员手动进行——前提是程序员能确认这样改写不会改变程序语义（例如确认 `f()` 是无副作用的纯函数）。

# 第三部分 处理器微体系结构

要有效地优化代码，必须理解现代处理器如何执行指令。

## 5.5 指令级并行（ILP）

现代处理器不是逐条顺序执行指令，而是利用多条功能单元**并行**执行多条指令。这称为**指令级并行（Instruction-Level Parallelism, ILP）**。

### 5.5.1 流水线与乱序执行

- **流水线**：多条指令在不同阶段同时进行（如第四章所述）
- **乱序执行**：指令可以不按原顺序完成，只要依赖关系满足

例如，对于指令序列：

```asm
addq  %rax, %rbx      # 结果存入 %rbx
movq  $10, %rcx       # 与 %rbx 无关
subq  %rax, %rdx      # 与 %rbx 无关
```

处理器可以先完成第二、三条指令，再等待第一条指令完成，只要最终的结果是正确的。

### 5.5.2 功能单元与容量

处理器含有多个**功能单元（Functional Unit）**，每个单元专门执行特定类型的操作。以 Intel Core i7 Haswell 为例：

| 单元     | 支持的操作             | 示例                   |
| -------- | ---------------------- | ---------------------- |
| ALU      | 整数加减、位运算、移位 | `addq`, `andq`, `xorq` |
| Multiply | 整数乘法、浮点乘法     | `imulq`, `vmulsd`      |
| Divide   | 整数和浮点除法         | `idivq`, `vdivsd`      |
| Load     | 内存读取               | `movq (%rax), %rbx`    |
| Store    | 内存写入               | `movq %rbx, (%rax)`    |

具体每个单元映射到哪个执行端口，属于硬件实现细节，此处从略，重点是理解处理器拥有**多个可并行工作的单元**这一事实。

### 5.5.3 性能指标

对于每个操作类型，处理器有一下重要的性能指标：

| 指标                       | 定义                                           | 含义                                             |
| -------------------------- | ---------------------------------------------- | ------------------------------------------------ |
| **延迟**（Latency）        | 指令完成所需的时钟周期数                       | 如果后续指令依赖该指令的结果，需要等待这么多周期 |
| **发射时间**（Issue Time） | 两个连续的同类型的运算之间需要的最小时钟周期数 | 指令                                             |
| **容量**（Capacity）       | 能够执行该运算的功能单元的数量                 | 在一个周期内可以发射的同类指令数                 |
| **吞吐量**（Throughput）   | 发射时间的倒数                                 | 处理器在单位时间内能处理的同类指令数量           |

### 5.5.4 典型操作的性能

延迟和吞吐因操作数据类型（整数 / 浮点）而异。以 Core i7 Haswell 为参考：

| Operation      | Integer: | Latency | Issue | Capacity | Floating point: | Latency | Issue | Capacity |
| -------------- | -------- | :-----: | :---: | :------: | --------------- | :-----: | :---: | :------: |
| Addition       |          |    1    |   1   |    4     |                 |    3    |   1   |    1     |
| Multiplication |          |    3    |   1   |    1     |                 |    5    |   1   |    2     |
| Division       |          |  3~30   | 3~30  |    1     |                 |  3~15   | 3~15  |    1     |

| Bound      | Integer: | $+$  | $\times$ | Floating point: | $+$  | $\times$ |
| ---------- | -------- | :--: | :------: | --------------- | :--: | :------: |
| Latency    |          | 1.00 |   3.00   |                 | 3.00 |   5.00   |
| Throughput |          | 0.50 |   1.00   |                 | 1.00 |   0.50   |

整数加法的延迟只有 1 周期、吞吐 0.5 周期/个（得益于处理器有多个 ALU，可同时发起两条加法），这与浮点加法的延迟（3 周期）有明显差距，区分两者是后续理解 [§5.12](#512-提高并行性parallelism) 中延迟界限和吞吐界限的基础。

**延迟与吞吐的区别**：

```asm
# 延迟示例：依赖链
vmulsd %xmm0, %xmm1, %xmm0    # 结果需要 5 周期
vaddsd %xmm0, %xmm2, %xmm0    # 必须等待前一条完成
```

此时 CPE ≈ 5（由延迟决定）。

```asm
# 吞吐示例：独立指令
vmulsd %xmm0, %xmm1, %xmm2    # 周期 1 发起
vmulsd %xmm3, %xmm4, %xmm5    # 周期 2 发起（与前一条无关）
vmulsd %xmm6, %xmm7, %xmm8    # 周期 3 发起
```

此时 CPE ≈ 1（由吞吐决定）。

## 5.6 关键路径

在有多条独立操作的代码中，**关键路径（Critical Path）** 是执行一组机器指令所需时钟周期数的一个下界。而依赖链最长的路径往往决定了程序的最低运行时间。

例如，对于累加操作：

```c
for(i = 0; i < n; i++){
    sum += data[i];  // 每次加法依赖前一个 sum
}
```

数据流图中只有一条链：

```
sum[0] -> (加法) -> sum[1] -> (加法) -> sum[2] -> ...
```

即使处理器有 4 个 ALU，它们也都在等待加法完成，无法并行工作。

# 第四部分 程序优化技术

现在我们介绍一系列实用的优化技术，并用向量运算的例子演示。

## 5.7 示例程序：向量累加

### 5.7.1 基础实现与性能度量

定义向量数据结构：

```c title="vec.h"
// 通过宏定义数据类型typedef int data_t或float data_t，便于切换 int、float 等
typedef struct {
    long len;
    data_t *data;
} vec_rec, *vec_ptr;


#define IDENT 0      // 加法的单位元
#define OP +         // 运算符

// 或者定义内积：
// #define IDENT 1
// #define OP *
```

最基础的实现：

```c
void combine1(vec_ptr v, data_t *dest){
    long i;
    *dest = IDENT;
    for(i = 0; i < vec_length(v); i++){
        data_t val;
        get_vec_element(v, i, &val);  // 获取第 i 个元素
        *dest = *dest OP val;
    }
}
```

测量的 CPE（单位：周期/元素）：

| Function | Method               | int $+$ | int $*$ | float $+$ | float $*$ |
| -------- | -------------------- | :-----: | :-----: | :-------: | :-------: |
| combine1 | Abstract unoptimized |  22.68  |  20.02  |   19.98   |   20.18   |
| combine1 | Abstract `-O1`       |  10.12  |  10.12  |   10.17   |   11.14   |

仅通过编译器的 `-O1` 优化就获得了 2 倍的性能提升。

### 5.7.2 问题分析

为什么 CPE 这么高？有两个原因：

1. **`vec_length(v)` 每次都被调用**：即使向量长度不变，编译器也不敢优化，因为它无法确定该函数是否有副作用（参考 [§5.4](#54-函数调用的副作用)）。
2. **目标指针 `dest` 每次迭代都被访问**：`*dest = *dest OP val` 涉及读-改-写，需要多次内存访问。

## 5.8 代码移动（Code Motion）

**技术**：将循环内不变的操作移出循环。

改进版本：

```c
void combine2(vec_ptr v, data_t *dest){
    long i;
    long len = vec_length(v);  // 移出循环
    *dest = IDENT;
    for(i = 0; i < len; i++){
        data_t val;
        get_vec_element(v, i, &val);
        *dest = *dest OP val;
    }
}
```

| Function | Method          | int $+$ | int $*$ | float $+$ | float $*$ |
| -------- | --------------- | :-----: | :-----: | :-------: | :-------: |
| combine1 | Abstract -O1    |  10.12  |  10.12  |   10.17   |   11.14   |
| combine2 | Move vec_length |  7.02   |  9.03   |   9.02    |   11.03   |

**性能提升**：~30%。虽然简单，但效果显著。

## 5.9 减少过程调用

正如前面提到的，仅仅将代码移动出循环就能带来显著的性能提升。那么，进一步减少循环体内的函数调用呢？

```c
void combine3(vec_ptr v, data_t *dest){
    long i;
    long len = vec_length(v);
    data_t *data = get_vec_start(v);    // 获取数据指针，避免每次调用 get_vec_element

    *dest = IDENT;
    for(i = 0; i < len; i++){
        *dest = *dest OP data[i];       // 直接访问数组元素
    }
}
```

| Function | Method              | int $+$ | int $*$ | float $+$ | float $*$ |
| -------- | ------------------- | :-----: | :-----: | :-------: | :-------: |
| combine2 | Move vec_length     |  7.02   |  9.03   |   9.02    |   11.03   |
| combine3 | Direct array access |  7.17   |  9.02   |   9.02    |   11.03   |

令人惊讶的是，性能几乎没有提升反而下降。显然，内循环中的其他操作形成了瓶颈，限制性能超过调用 `get_vec_element`

## 5.10 消除不必要的内存引用

**问题**：在 `combine2` 中，`*dest = *dest OP val` 每次迭代都涉及对内存的一次读和一次写。

根本原因在于：只要累加结果存放在 `*dest` 指向的内存中，编译器就无法排除它与 `data` 数组发生别名重叠的可能性（参考 §5.3）。出于安全考虑，编译器不敢把累加值长期保留在寄存器里，被迫在每次迭代都重新从内存读取、计算后再写回内存，这就在循环中引入了一条贯穿始终的内存依赖链。即便处理器支持存储-加载转发（store-forwarding）缓解这一访问的部分延迟，这条依赖链本身依然是性能瓶颈的根源。

**解决**：用本地变量 `acc` 缓存累加结果，由于 `acc` 是局部变量、不可能被外部指针引用，编译器可以放心地将它一直留在寄存器中，最后才写回内存：

```c
void combine4(vec_ptr v, data_t *dest){
    long i;
    long len = vec_length(v);
    data_t *data = get_vec_start(v);  // 直接获取数据指针
    data_t acc = IDENT;               // 本地累加变量
    for(i = 0; i < len; i++){
        acc = acc OP data[i];         // 只用寄存器
    }
    *dest = acc;                      // 循环后写回一次
}
```

| Function | Method            | int $+$ | int $*$ | float $+$ | float $*$ |
| -------- | ----------------- | :-----: | :-----: | :-------: | :-------: |
| combine2 | Move vec_length   |  7.02   |  9.03   |   9.02    |   11.03   |
| combine3 | Compiled -O2      |  1.60   |  3.01   |   3.01    |   5.01    |
| combine4 | Accumulate in acc |  1.27   |  3.01   |   3.01    |   5.01    |

**性能提升**：2~3 倍！这是因为大大消除了循环体内的内存读写，依赖链从"内存中的 `*dest`"变成了"寄存器中的 `acc`"，后者的延迟远低于前者，实现了与`-O2`优化相当的性能。

## 5.11 循环展开（Loop Unrolling）

**思想**：每次迭代处理多个元素，减少循环开销（循环变量更新、条件判断等）。

**$2 \times 1$ 展开**（每次处理 2 个元素，1 个累加变量）：

```c
void combine5(vec_ptr v, data_t *dest){
    long i;
    long len = vec_length(v);
    data_t *data = get_vec_start(v);
    data_t acc = IDENT;
    // 每次处理两个元素
    for(i = 0; i < len - 1; i += 2){
        acc = (acc OP data[i]) OP data[i+1];
    }
    // 处理剩余元素
    if(i < len)
        acc = acc OP data[i];
    *dest = acc;
}
```

| Function | Method                 | int $+$ | int $*$ | float $+$ | float $*$ |
| -------- | ---------------------- | :-----: | :-----: | :-------: | :-------: |
| combine4 | Accmulate in acc       |  1.27   |  3.01   |   3.01    |   5.01    |
| combine5 | $2 \times 1$ unrolling |  1.01   |  3.01   |   3.01    |   5.01    |

**性能提升**：有限。对于加法有轻微改善，但乘法没有改善。原因是**仍然存在依赖链**：

```
acc = (acc OP data[i]) OP data[i+1]
      
      依赖前面的 acc
```

第二次操作必须等待第一次完成。

事实上，如果画出数据流图，会发现仍然与`combine4` 的数据流图类似（简化后），只是每次迭代处理两个元素而已，并没有打破依赖链的限制。

## 5.12 提高并行性（Parallelism）

**关键洞察**：如果能创建**多条独立的依赖链**，处理器可以并行执行，打破延迟的限制。

### 5.12.1 多累积变量

**$2 \times 2$ 展开**（每次处理 2 个元素，2 个独立的累加变量）：

```c
void combine6(vec_ptr v, data_t *dest){
    long i;
    long len = vec_length(v);
    data_t *data = get_vec_start(v);
    data_t acc0 = IDENT, acc1 = IDENT;  // 两个独立累加变量

    for(i = 0; i < len - 1; i += 2){
        acc0 = acc0 OP data[i];        // 路径 1
        acc1 = acc1 OP data[i+1];      // 路径 2（独立）
    }
    if(i < len)
        acc0 = acc0 OP data[i];

    *dest = acc0 OP acc1;  // 合并结果
}
```

### 5.12.2 重新结合变换（Reassociation Transformation）

考虑这样的代码变换：

```c
// 方案 A：连续结合 -> combine5
acc = (acc OP data[i]) OP data[i+1];

// 方案 B：先计算 data，再与 acc 合并 -> combine7
acc = acc OP (data[i] OP data[i+1]);
```

这两个看似等价，但在流水线上的表现完全不同。

| Function | Method                  | int $+$ | int $*$ | float $+$ | float $*$ |
| -------- | ----------------------- | :-----: | :-----: | :-------: | :-------- |
| combine5 | $2 \times 1$ unrolling  |  1.01   |  3.01   |   3.01    | 5.01      |
| combine6 | $2 \times 2$ unrolling  |  0.81   |  1.51   |   1.51    | 2.51      |
| combine7 | $2 \times 1a$ unrolling |  1.01   |  1.51   |   1.51    | 2.51      |

在**方案 A** 中，`acc OP data[i]` 必须先算完，才能进行下一步 `... OP data[i+1]`，依赖链长度等于 2 倍延迟。

在**方案 B** 中，`data[i] OP data[i+1]` 不依赖 `acc`，可以独立计算；只有最后一步 `acc OP (...)` 才依赖原来的 `acc`，依赖链长度只有 1 倍延迟，外加一个可以与之并行执行的中间计算。

但实际上，重新结合的效果取决于编译器的具体安排。通常，与**多累积变量**一起使用效果最好。

## 5.13 一些限制因素

### 5.13.1 寄存器溢出（Register Spilling）

从数据可以看出，继续展开循环的收益是有限的：

| Function       | Method                   | int $+$ | int $*$ | float $+$ | float $*$ |
| -------------- | ------------------------ | :-----: | :-----: | :-------: | :-------- |
| combine6       | $2 \times 2$ unrolling   |  0.81   |  1.51   |   1.51    | 2.51      |
|                | $10 \times 10$ unrolling |  0.55   |  1.00   |   1.01    | 0.52      |
|                | $20 \times 20$ unrolling |  0.83   |  1.03   |   1.02    | 0.68      |
| **Throughput** |                          |  0.50   |  1.00   |   1.00    | 0.50      |

可以看到，随着展开程度继续增加，CPE 不再持续下降，部分结果甚至回升。

现代x86-64 处理器有 16 个寄存器，并可以使用 16 个 YMM 寄存器来保存浮点数。一旦循环变量的数量超过了可用寄存器的数量，程序就必须在栈上分配一些变量。

一旦发生**寄存器溢出**，额外的内存读写会抵消多累积变量带来的收益。幸运的是，在 x86-64 上有足够多的寄存器，大多数循环通常会先接近处理器的吞吐界限，而不是先受到寄存器数量的限制。

### 5.13.2 分支预测与预测错误处罚

循环体中如果包含数据相关的条件分支，就会引入第四章所讨论的**分支预测**开销：一旦预测失败，流水线中已经取入的指令需要被取消，从而产生额外的停顿周期。

当条件两侧都是简单且无副作用的计算时，编译器往往会生成条件传送指令 `cmovcc`，而不是条件跳转。这样可以避免分支预测失败带来的惩罚。对于性能敏感的循环，如果某个 `if` 语句的数据相关性很强且两侧分支都很短，将其改写为不依赖分支的等价形式，通常更有利于编译器生成条件传送代码。

对参考机而言，一次预测错误的代价约为 19 个时钟周期，因此分支行为是否可预测会显著影响性能。虽然没有适用于所有程序的统一规则，但通常可以遵循以下原则：

1. 不要过度担心可预测的分支
   - 现代处理器的分支预测器能够很好地识别具有规律性的模式和长期趋势。循环结束条件、计数器递减这类分支通常具有高度可预测性，真正产生较大开销的往往是少数预测失败。
   - 例如，执行边界检测所需的额外计算可以与合并操作并行执行。 处理器能够预测这些分支的结果， 所以这些求值都不会对形成程序执行中关键路径的指令的取指和处理产生太大的影响。

2. 尽量使用适合条件传送的代码形式
   - 对于无法稳定预测的条件判断，如果编译器能够将其翻译为条件传送而不是条件跳转，就可以避免分支错误带来的停顿。程序员虽然不能直接控制机器指令选择，但可以通过改写表达式，让编译器更容易采用这种实现方式。

# 第五部分 内存性能优化

之前的优化主要针对 ALU。内存访问则是另一个瓶颈。

## 5.14 加载延迟与链表遍历

考虑遍历链表的代码：

```c
typedef struct ELF {
    struct ELF *next;
    long data;
} list_ele, *list_ptr;

long list_len(list_ptr ls){
    long len = 0;
    while(ls){
        len++;
        ls = ls->next;  // 指针依赖
    }
    return len;
}
```

对应的关键汇编代码：

```asm
.L3
    addq    $1, %rax              # len++
    movq    (%rdi), %rdi          # ls = ls->next （从内存加载）
    testq   %rdi, %rdi            # 检测 ls 是否为 NULL
    jne     .L3
```

**性能瓶颈**：后一次加载依赖前一次加载的结果（必须先拿到 `next` 指针才能知道下一个加载的地址），无法通过转发或并行手段绕开，CPE 等于 L1 缓存的加载延迟，约 **4 周期**。这与 [§5.5.4](#554-典型操作的性能) 中"加载延迟 4 周期"的数据一致。

## 5.15 存储与加载的相互影响

**写-读相关（Write-Read Dependence）**：存储操作可能影响后续的加载。

```c
void write_read(long *src, long *dst, long n){
    long val = 0;
    while(n--){
        *dst = val;
        val = *src + 1;
    }
}
```

性能差异：

- 若 `src` 和 `dst` **不重叠**：CPE ≈ 1.3（存储不产生数据依赖，加载可以提前发起）
- 若 `src` 和 `dst` **重叠**（例如 `write_read(&a[0], &a[0], n)`）：CPE 显著上升，量级接近存储转发延迟本身（约 5 周期左右）

原因是处理器必须保守地假设加载可能读取刚刚存储的值，所以必须等待存储完成、通过存储转发机制取得数据后才能继续，这条等待链直接体现在 CPE 上。
