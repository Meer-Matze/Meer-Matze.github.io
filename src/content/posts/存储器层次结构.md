# 第一部分 存储技术

## 6.1 随机访问存储器（RAM）

**随机访问存储器（Random Access Memory，RAM）** 分为两类：SRAM 和 DRAM。

### 6.1.1 SRAM 与 DRAM

| 特性         |            SRAM             |            DRAM            |
| ------------ | :-------------------------: | :------------------------: |
| 存储原理     | 双稳态触发器（6 晶体管/位） | 电容（1 晶体管+1 电容/位） |
| 相对访问时间 |          $1\times$          |         $10\times$         |
| 是否需要刷新 |             否              |             是             |
| 对干扰敏感   |             否              |             是             |
| 相对成本     |        $1000\times$         |         $1\times$          |
| 主要用途     |          高速缓存           |            主存            |

SRAM 和 DRAM 都是 **易失性（volatile）** 存储器，断电后数据均会丢失。

两者的区别在于：

- DRAM 使用电容存储电荷，电容会漏电，因此即使在通电状态下也需要 **周期性刷新** ，否则数据丢失
- SRAM 使用六晶体管双稳态单元，只要保持通电就能稳定保持数据，无需刷新——"Static"的含义正在于此。

### 6.1.2 传统 DRAM 结构

DRAM 芯片中的存储单元被组织为 $d$ 个**超单元（supercell）**，每个超单元由 $w$ 个 DRAM 单元组成（通常 $w = 8$）。一个 $d \times w$ 的 DRAM 芯片总共存储 $d \cdot w$ 位信息。超单元进一步被组织成 $r \times c$ 的矩形阵列，其中 $r \cdot c = d$。

读取数据时，内存控制器通过引脚向 DRAM 发送地址信号。访问流程如下：

1. 通过行地址选择某一行的超单元
2. 该行全部数据被复制到**行缓冲区（row buffer）**
3. 通过列地址从行缓冲区中选择所需的那个超单元

这种**行列分时寻址**的设计虽然增加了访问延迟，但有效减少了引脚数量。

### 6.1.3 内存模块

DRAM 芯片封装在**内存模块（memory module）** 中。例如，用 8 片 64 Mbit 的 $8\mathrm{M} \times 8$ DRAM 芯片组成 64 MB 的内存模块。要读出一个字节（8 位），内存控制器同时向 8 个芯片的同一位地址发送列地址，每个芯片输出 1 位，组合起来即为一个字节。

### 6.1.4 增强的 DRAM

|   类型    | 特点                                                                                                                         |
| :-------: | ---------------------------------------------------------------------------------------------------------------------------- |
| FPM DRAM  | 允许对同一行的连续访问直接从行缓冲区得到服务，无需重新激活该行                                                               |
| EDO DRAM  | 将数据输出有效的时间延长，使内存控制器可以在当前列数据仍在总线上时就开始下一次列地址选通，实现同一行内相邻列访问的流水线重叠 |
|   SDRAM   | 使用时钟信号同步所有操作                                                                                                     |
| DDR SDRAM | 在时钟上升沿**和**下降沿均传输数据（如 DDR4、LPDDR4）                                                                        |

## 6.2 磁盘存储

### 6.2.1 机械磁盘（HDD）

**机械磁盘（Hard Disk Drive，HDD）** 使用旋转盘片存储数据，盘片表面涂有磁性记录材料。

**物理结构**：

- 盘片表面划分为同心圆状的**磁道（track）**
- 每个磁道划分为若干**扇区（sector）**，通常每扇区 512 字节或 4 KB
- 每个盘面有一个独立的**读/写头（read/write head）**，所有磁头安装在同一个**传动臂（actuator arm）** 上

**访问时间**由三部分组成：

| 时间组成                           | 说明                                                                           |            典型值            |
| ---------------------------------- | ------------------------------------------------------------------------------ | :--------------------------: |
| **寻道时间**（seek time）          | 磁头移动到目标磁道                                                             |     $3 \sim 9\text{ ms}$     |
| **旋转延迟**（rotational latency） | 盘片旋转到目标扇区；<br>$T_{avg~rot} =\frac{1}{2} \times \frac{1}{\text{RPM}}$ | 约 $4\text{ ms}$（7200 RPM） |
| **传输时间**（transfer time）      | 扇区转过磁头下方；<br>$T_{transfer} = \frac{\text{平均扇区数}}{\text{RPM}}$    |     约 $0.02\text{ ms}$      |

**容量单位**：磁盘厂商通常使用十进制单位（1 GB = $10^9$ 字节），与二进制单位（1 GiB = $2^{30}$ 字节）不同，中间的 "i" 表示二进制前缀。

**逻辑块地址（LBA）**：磁盘驱动器将物理结构抽象为线性地址空间，每个 LBA 对应一个扇区。驱动器负责将 LBA 映射为三元组（磁道，扇区，盘面），并通过调度算法合并相邻请求以减少寻道时间。

### 6.2.2 固态硬盘（SSD）

**固态硬盘（Solid State Drive，SSD）** 基于 NAND 闪存，无机械部件。

**层次结构**：

```
闪存芯片 → 逻辑单元(Die) → 面(Plane) → 块(Block) → 页(Page)
```

- 数据以**页**为单位读写
- 数据以**块**为单位擦除
- 闪存的物理特性决定了数据只能将 1 改为 0（写），不能将 0 改为 1；因此写入前必须先擦除（全部置 1）

**FTL（Flash Translation Layer）**：负责地址映射、垃圾回收和**磨损均衡（wear leveling）**——将擦除操作均匀分布到所有块上，避免个别块过早报废。

**与 HDD 的对比**：

| 特性     |        HDD        |       SSD        |
| -------- | :---------------: | :--------------: |
| 随机访问 | 慢（需寻道+旋转） | 快（无机械部件） |
| 抗震性   |        差         |        好        |
| 功耗     |       较高        |       较低       |
| 价格/GB  |        低         |        高        |
| 容量上限 |        大         |     相对较小     |

## 6.3 存储技术的发展趋势

1. 不同存储技术在价格和性能之间存在不同的折中关系
2. 不同存储技术的价格和性能属性以**截然不同的速度**变化
3. DRAM 和磁盘的性能改善速度**落后于 CPU**——这导致了"存储墙"问题，使存储器层次结构成为现代计算机性能的关键

建立对各层次访问延迟数量级的直觉对理解后续内容至关重要：

| 存储层次        | 典型访问延迟（时钟周期） | 典型绝对延迟 |
| --------------- | :----------------------: | :----------: |
| L1 高速缓存     |         ~4 周期          |    ~1 ns     |
| L2 高速缓存     |         ~10 周期         |    ~3 ns     |
| L3 高速缓存     |         ~40 周期         |    ~12 ns    |
| 主存（DRAM）    |        ~200 周期         |    ~60 ns    |
| 固态硬盘（SSD） |        ~数万周期         |   ~100 μs    |
| 机械磁盘（HDD） |       ~数千万周期        |    ~10 ms    |

一个形象的类比：如果 L1 缓存命中需要 1 秒，那么访问主存需要约 1 分钟，访问机械磁盘则需要数周。这就是为什么存储器层次结构如此重要——不同层次之间的延迟差距是数量级级别的。

# 第二部分 局部性

## 6.4 局部性原理

一个编写良好的计算机程序常常具有良好的**局部性（locality）**：它们倾向于引用邻近于最近引用过的数据项的数据项，或者最近引用过的数据项本身。这一观察称为**局部性原理（principle of locality）**，对硬件和软件系统的设计有着深远影响。

### 6.4.1 时间局部性与空间局部性

| 类型                                | 定义                                     | 典型场景                 |
| ----------------------------------- | ---------------------------------------- | ------------------------ |
| **时间局部性**（temporal locality） | 被访问过的数据在不久后很可能再次被访问   | 循环中的累加变量、计数器 |
| **空间局部性**（spatial locality）  | 被访问数据附近的数据在不久后很可能被访问 | 数组的顺序遍历           |

具有良好局部性的程序运行更快，因为它们能更充分地利用高速缓存。

### 6.4.2 程序数据引用的局部性示例

```c
int sumarrayrows(int a[M][N]) {
    int i, j, sum = 0;
    for (i = 0; i < M; i++)
        for (j = 0; j < N; j++)
            sum += a[i][j];
    return sum;
}
```

- `sum`：时间局部性好（每次迭代都被访问），无空间局部性
- 数组 `a`：空间局部性好（按行优先顺序访问相邻元素），但无时间局部性（每个元素只访问一次）

若交换循环顺序：

```c
int sumarraycols(int a[M][N]) {
    int i, j, sum = 0;
    for (j = 0; j < N; j++)
        for (i = 0; i < M; i++)
            sum += a[i][j];
    return sum;
}
```

由于 C 语言中的二维数组按**行优先（row-major order）** 存储，`i` 的变化会导致跨行访问，空间局部性显著下降。

### 6.4.3 取指令的局部性

程序指令也存放在内存中，取指令同样具有局部性：

- **时间局部性**：循环反复执行同一段代码
- **空间局部性**：程序通常顺序执行指令

# 第三部分 存储器层次结构与缓存

## 6.5 存储器层次结构

典型的存储器层次结构如下：

|  特征  | 层次         | 存储技术           | 保存内容            |
| :----: | :----------- | :----------------- | ------------------- |
|  更小  | 寄存器       | —                  | 从高速缓存取出的字  |
|  更快  | L1 高速缓存  | SRAM               | L2 高速缓存的缓存行 |
|  更贵  | L2 高速缓存  | SRAM               | L3 高速缓存的缓存行 |
|        | L3 高速缓存  | SRAM               | 主存的缓存行        |
|  更大  | 主存         | DRAM               | 本地磁盘的磁盘块    |
|  更慢  | 本地二级存储 | 磁盘               | 远程服务器上的文件  |
| 更便宜 | 远程二级存储 | 分布式文件系统/Web | —                   |

层次结构的特点是：越往上速度越快、容量越小、单位成本越高；越往下则相反。

## 6.6 缓存基本原理

**高速缓存（cache）** 是一个小而快速的存储设备，作为更大、更慢的存储设备的缓冲区域。

### 6.6.1 缓存命中与不命中

第 $k+1$ 层的存储器被划分为连续的数据块（block）。数据以块为单位在相邻两层间复制。当程序需要第 $k+1$ 层的数据对象 $d$ 时：

1. **缓存命中（cache hit）**：$d$ 恰好已在第 $k$ 层中，直接使用
2. **缓存不命中（cache miss）**：第 $k$ 层中不包含 $d$，需要从第 $k+1$ 层将包含 $d$ 的块加载到第 $k$ 层；若第 $k$ 层已满，需**替换（replace）/驱逐（evict）** 一个现有块

### 6.6.2 平均内存访问时间（AMAT）

缓存性能可以用**平均内存访问时间（Average Memory Access Time，AMAT）** 量化：

$$\text{AMAT} = \text{命中时间} + \text{不命中率} \times \text{不命中惩罚}$$

其中：

- **命中时间**：从缓存中直接读到数据所需的时间
- **不命中率**：访问请求中导致缓存不命中的比例
- **不命中惩罚**：发生不命中后，从下一层存储器取回数据所需的额外时间

这个公式是理解缓存各参数对性能影响的量化工具。例如，L1 命中时间为 4 周期，不命中率为 2%，访问 L2 的惩罚为 10 周期，则 $\text{AMAT} = 4 + 0.02 \times 10 = 4.2$ 周期。可以看出，即使不命中率只有 2%，命中时间仍主导了总延迟；但若不命中惩罚增大到访问 DRAM 的 200 周期，则 $\text{AMAT} = 4 + 0.02 \times 200 = 8$ 周期——不命中惩罚的贡献不可忽视。

### 6.6.3 不命中的类型

| 类型                                        | 原因                     | 说明                           |
| ------------------------------------------- | ------------------------ | ------------------------------ |
| **冷不命中**（cold miss / compulsory miss） | 缓存初始为空             | 第一次访问某个数据块必然不命中 |
| **容量不命中**（capacity miss）             | 缓存大小不足以容纳工作集 | 程序访问的数据超过缓存容量     |
| **冲突不命中**（conflict miss）             | 多个地址映射到同一缓存组 | 由于映射策略限制导致频繁替换   |

### 6.6.4 替换策略

当缓存已满时，需要选择被替换的块，这个块称为**牺牲块（victim block）**。常见的替换策略包括：

| 策略                                         | 说明                     |
| -------------------------------------------- | ------------------------ |
| **随机替换**（random）                       | 随机选择一个块替换       |
| **最近最少使用**（LRU, Least Recently Used） | 替换最长时间未被使用的块 |
| **先入先出**（FIFO）                         | 替换最早进入的块         |

LRU 通常在组相联缓存中使用，能较好地利用时间局部性，但实现成本较高。

# 第四部分 高速缓存存储器

## 6.7 高速缓存的组织结构

### 6.7.1 通用结构

考虑一个 $m$ 位地址的系统，共 $M = 2^m$ 个地址。高速缓存的组织可用元组 $(S, E, B, m)$ 描述：

| 参数      |            含义             |
| --------- | :-------------------------: |
| $S = 2^s$ |   高速缓存组（set）的数量   |
| $E$       | 每组的高速缓存行（line）数  |
| $B = 2^b$ | 每个数据块（block）的字节数 |
| $m$       |        物理地址位数         |

高速缓存容量 $C = S \times E \times B$（仅计算数据块大小，不含标记位和有效位）。

每一行包含：

- **有效位（valid bit）**：1 位，标识该行是否包含有意义的数据
- **标记位（tag）**：$t$ 位，$t = m - (b + s)$
- **数据块（block）**：$B$ 字节

### 6.7.2 地址划分

$m$ 位地址被划分为三个部分：

| $t$ 位（标记） | $s$ 位（组索引） | $b$ 位（块偏移） |
| :------------: | :--------------: | :--------------: |
|      tag       |    set index     |   block offset   |

访问流程（三步）：

1. **组选择**：使用组索引位选择高速缓存中的一个组
2. **行匹配**：在组内检查每个行的有效位和标记位，若匹配则命中
3. **字选择**：使用块偏移位从数据块中选择所需的字节

### 6.7.3 为什么使用中间位作为索引

高速缓存使用地址的**中间位**（而非高位）作为组索引。考虑一个具有 4 个组（每组 1 行）的缓存和一个连续数组的访问场景：

- **若使用高位索引**：连续数组元素的高位相同，会被映射到同一个组，导致频繁替换，空间局部性被破坏
- **若使用中间位索引**：连续元素被均匀映射到不同组，各组都能得到利用，空间局部性得以发挥

采用中间位索引可以更好地利用空间局部性，将连续地址分散到不同缓存组中，减少冲突不命中。

## 6.8 高速缓存的映射方式

根据每组的高速缓存行数 $E$，高速缓存分为三类。

### 6.8.1 直接映射（Direct-Mapped Cache）

**定义**：$E = 1$，每组只有一行。

给定地址后，组索引唯一确定组，组内唯一行要么命中要么不命中。若不命中，直接替换该行。

**抖动（thrashing）** 示例：

```c
float dotprod(float x[8], float y[8]) {
    int i;
    float sum = 0.0;
    for (i = 0; i < 8; i++)
        sum += x[i] * y[i];
    return sum;
}
```

若 `x` 和 `y` 的地址映射到同一个缓存组，每次访问 `x[i]` 会替换掉 `y[i]` 的缓存行，反之亦然，导致抖动。解决方法之一是调整数组长度或对齐方式，使二者映射到不同组。

### 6.8.2 组相联（Set-Associative Cache）

**定义**：$1 < E < \frac{C}{B}$，称为 $E$**路组相联高速缓存**。

组选择和字选择与直接映射一致，但行匹配时需要遍历组内所有行（最多 $E$ 行），找到有效位为 1 且标记匹配的行即为命中。

不命中时，组内有多个候选行可供替换，因此可以使用更复杂的替换策略（如 LRU），减少冲突不命中的概率。

### 6.8.3 全相联（Fully Associative Cache）

**定义**：$S = 1$，$E = \frac{C}{B}$，整个缓存只有一个组。

此时地址只划分为标记位和块偏移位，无需组选择。行匹配时需要搜索所有行。

全相联缓存冲突不命中最少，但硬件实现复杂、成本高。通常仅用于小型缓存，如 TLB（翻译后备缓冲器）。

## 6.9 写入策略

读取缓存相对简单，写入则更为复杂，涉及命中与不命中两种情形。

### 写命中

| 策略                      | 做法                                                                                         |                   优缺点                   |
| ------------------------- | -------------------------------------------------------------------------------------------- | :----------------------------------------: |
| **直写**（write-through） | 数据**同时**写入缓存和下一级存储器                                                           | 实现简单，但每次写操作都需访问低速的下一级 |
| **写回**（write-back）    | 数据**只**写入缓存；使用**脏位（dirty bit）** 标记被修改的行，只有在该行被替换时才写回下一级 |        减少写传输次数，但管理更复杂        |

### 写不命中

| 策略                              | 做法                         |              适用场景              |
| --------------------------------- | ---------------------------- | :--------------------------------: |
| **写分配**（write-allocate）      | 将块加载到缓存后再写入       |  局部性好的程序（后续访问可命中）  |
| **非写分配**（no-write-allocate） | 直接写入下一级，不加载到缓存 | 局部性差的程序（避免浪费缓存空间） |

**常见组合**：写回 + 写分配；直写 + 非写分配。

## 6.10 真实层次结构：Intel Core i7

以 Intel Core i7 为例：

```
CPU 芯片
├── 核心 0
│   ├── L1 i-cache（指令缓存）
│   ├── L1 d-cache（数据缓存）
│   └── L2 统一缓存（指令+数据）
├── 核心 1
│   └── ...
├── 核心 2
│   └── ...
└── 核心 3
    └── ...
└── L3 统一缓存（所有核心共享）
```

- L1 和 L2 为每个核心私有
- L3 为所有核心共享
- 所有 SRAM 缓存均在 CPU 芯片上

Core i7（Haswell 某型号）的缓存参数（具体数值因型号而异）：

| 缓存        | 类型      | 大小（C） | 相联度（E） | 块大小（B） | 组数（S） |
| ----------- | --------- | :-------: | :---------: | :---------: | :-------: |
| L1 i-cache  | 指令缓存  |   32 KB   |      8      |    64 B     |    64     |
| L1 d-cache  | 数据缓存  |   32 KB   |      8      |    64 B     |    64     |
| L2 统一缓存 | 指令+数据 |  256 KB   |      8      |    64 B     |    512    |
| L3 统一缓存 | 共享      |   8 MB    |     16      |    64 B     |   8192    |

## 6.11 高速缓存参数的性能影响

### 6.11.1 缓存大小

- **较大缓存**：提高命中率，但增大访问时间（使大存储器运行更快更难）
- **较小缓存**：降低命中时间
- 这就是 L1（最小最快）→ L2 → L3（最大最慢）的层次设计原因

### 6.11.2 块大小

- **较大块**：利用空间局部性提高命中率；但行数减少，损害时间局部性；且不命中惩罚更大（传输时间更长）
- **较小块**：行数增多，时间局部性好，但空间局部性利用不足
- 现代处理器（如 Core i7）的典型折中：**64 字节**

### 6.11.3 相联度（$E$）

- **高相联度**：降低冲突不命中概率；但成本高、实现复杂（更多标记位、LRU 状态位、控制逻辑），命中时间增加
- **低相联度**：命中时间短，但易发生冲突不命中
- 实际折中：L1/L2 常用 8 路，L3 常用 16 路（不命中惩罚更大，值得更高的相联度）

### 6.11.4 写策略

- **直写**：实现简单，可用写缓冲区加速；读不命中开销小（不触发内存写）
- **写回**：减少写传输次数，为 DMA I/O 设备留出更多带宽；越往层次下层，写回的优势越明显

# 第五部分 编写高速缓存友好的代码

## 6.12 基本原则

局部性好的程序不命中率更低，运行更快。编写**高速缓存友好（cache-friendly）**代码时应遵循以下原则：

1. **把精力集中在核心循环**：最常见的情况应运行得最快
2. **减少循环内部的缓存不命中**

### 6.12.1 步长为 1 的访问模式

```c
int sumvec(int v[N]) {
    int i, sum = 0;
    for (i = 0; i < N; i++)
        sum += v[i];
    return sum;
}
```

- 局部变量 `sum` 和 `i` 具有时间局部性，编译器会将其放入寄存器
- 向量 `v` 采用**步长为 1** 的访问模式

若缓存块大小为 $B$ 字节，字长为 $w$ 字节，步长为 $k$ 个字，则平均每次迭代的缓存**不命中率**为：

$$\min\left(1, \frac{w \times k}{B}\right)$$

当 $k = 1$ 时不命中率最低，因此步长为 1 的引用是**缓存友好**的。注意当 $k$ 较小时，结果小于 1，这是一个概率值而非整数计数——例如对于 4 字节的 `int` 和 64 字节的块，$k=1$ 时不命中率为 $4/64 = 1/16$，即每 16 次访问才发生一次不命中。

**示例**（块大小 = 4 个字，冷缓存）：

| `v[i]` |  0  |  1  |  2  |  3  |  4  |  5  |  6  |  7  |
| :----: | :-: | :-: | :-: | :-: | :-: | :-: | :-: | :-: |
| 命中？ | ❌  | ✅  | ✅  | ✅  | ❌  | ✅  | ✅  | ✅  |

- `v[0]` 不命中，加载块 `v[0]~v[3]`，后续三次命中
- `v[4]` 不命中，加载块 `v[4]~v[7]`，后续三次命中
- 不命中率 = 2/8 = 25%，命中率 = 75%

### 6.12.2 二维数组的行列访问对比

行优先访问（`sumarrayrows`）与列优先访问（`sumarraycols`）的不命中率差异可以量化。设块大小为 $B$ 字节，元素大小为 $w$ 字节（如 `int` 为 4 字节）：

**行访问**（步长为 1）：每 $B/w$ 个元素发生一次不命中，不命中率 = $w/B$。以 64 字节块、4 字节 int 为例，不命中率 = $4/64 = 1/16 \approx 6\%$。

**列访问**（步长为 N）：每次访问都跳到下一行，步长为 $N$ 个元素。若矩阵足够大（$N \times w > C$，不能完整放入缓存），则每次访问都不命中，不命中率趋近于 1（100%）。

两者相差数十倍乃至更多，这正是 `sumarrayrows` 比 `sumarraycols` 快得多的根本原因，而不仅仅是"空间局部性好坏"的定性描述。

### 6.12.3 总结

- **局部变量的反复访问是好的**（时间局部性）
- **步长为 1 的访问模式是好的**（空间局部性）
- **按列访问二维数组是差的**（每次访问都可能不命中，不命中率趋近于 1）

## 6.13 矩阵乘法与分块优化

矩阵乘法是说明缓存影响的经典案例。

### 6.13.1 朴素实现的缓存瓶颈

标准的 $N \times N$ 矩阵乘法 $C = A \times B$：

```c
for (i = 0; i < N; i++)
    for (j = 0; j < N; j++)
        for (k = 0; k < N; k++)
            C[i][j] += A[i][k] * B[k][j];  /* B 按列访问 */
```

分析三个矩阵的访问模式：

- `A[i][k]`：固定 `i`，`k` 递增，**步长为 1**，空间局部性好
- `B[k][j]`：固定 `j`，`k` 递增，**步长为 N**，按列访问，空间局部性差
- `C[i][j]`：在最内层循环不变，时间局部性极好（可保留在寄存器中）

当 $N$ 足够大时，`B` 的每次访问几乎都不命中，不命中率趋近于 1。整个乘法的性能瓶颈就在这里。

### 6.13.2 分块（Blocking / Loop Tiling）

**核心思想**：将矩阵划分为大小为 $b \times b$ 的**子块（block）**，使三个子块能同时装入缓存；在子块内部完成局部乘法后再移动到下一个子块，让数据在被驱逐出缓存之前得到充分复用。

```c
for (i = 0; i < N; i += b)
    for (j = 0; j < N; j += b)
        for (k = 0; k < N; k += b)
            /* 对子块做乘法累加 */
            for (ii = i; ii < i + b; ii++)
                for (jj = j; jj < j + b; jj++)
                    for (kk = k; kk < k + b; kk++)
                        C[ii][jj] += A[ii][kk] * B[kk][jj];
```

**效果分析**：

在朴素实现中，对于矩阵 B 的每次访问，不命中率为 1（每列都不命中）。分块后，子块 B 的 $b \times b$ 个元素可以完整驻留在缓存中，在 $b$ 次外层迭代中被反复使用，不命中率降至约 $1/b$。

块大小 $b$ 的选择原则是：三个 $b \times b$ 子块合计大小不超过缓存容量 $C$，即 $3b^2 w \leq C$，由此得 $b \leq \sqrt{C / (3w)}$。以 L1 缓存 32 KB、`double`（8 字节）为例，$b \leq \sqrt{32768 / 24} \approx 37$，通常取 32 或 64 的整数倍对齐。

**分块是缓存友好代码技术中最重要的一种**：它不改变算法的计算量，只通过重排访问顺序来改变数据的时间局部性，能使矩阵乘法的实际性能提升数倍乃至十倍以上。

> [!note]
> 分块使得代码更难阅读和理解。由于这个原因，它最适合于优化编译器或者频繁执行的库函数。

由于 Core i7 有完善的预取硬件，分块不会提高矩阵乘在 Core i7 上的性能。
