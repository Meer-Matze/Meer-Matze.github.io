# 第一部分 处理器基础

## 4.1 处理器执行模型概述

处理器执行程序的本质就是按顺序取出指令并执行。一个现代处理器的执行周期包括以下几个关键步骤：

| 步骤    | 英文       | 说明                                             |
| ------- | ---------- | ------------------------------------------------ |
| 取指    | Fetch      | 从内存按程序计数器（PC）取出一条指令             |
| 译码    | Decode     | 解析指令的操作码和操作数，从寄存器文件读取操作数 |
| 执行    | Execute    | 进行计算（ALU 操作、地址计算等）                 |
| 访存    | Memory     | 读取或写入内存（如果指令需要）                   |
| 写回    | Write Back | 将结果写回寄存器文件                             |
| 更新 PC | Update PC  | PC 指向下一条指令                                |

这六个阶段是**顺序处理器（SEQ, Sequential Processor）**的划分方式：一条指令完全执行完毕后，下一条指令才能开始。现代处理器采用**流水线**技术，将这六个阶段压缩、重组为五个流水线阶段（见[第二部分](#第二部分-流水线处理)），使多条指令可以同时执行。

## 4.2 Y86-64 简介

Y86-64 是 x86-64 的简化教学版本。它包含：

- **15 个通用寄存器**：`%rax`, `%rbx`, `%rcx`, `%rdx`, `%rsi`, `%rdi`, `%rsp`, `%rbp`, `%r8`, `%r9`, `%r10`, `%r11`, `%r12`, `%r13`, `%r14`
- **一个程序计数器**（PC）：指向当前执行的指令地址
- **内存**：地址空间为 64 位，可存储指令和数据
- **条件码寄存器**：记录最后一次运算的结果特性

条件码包括三个标志位（Y86-64 不含 CF，因为没有无符号算术指令和跳转指令）：

| 标志位              | 含义                   |
| ------------------- | ---------------------- |
| ZF（Zero Flag）     | 最后运算结果为 0       |
| SF（Sign Flag）     | 最后运算结果为负数     |
| OF（Overflow Flag） | 最后运算发生有符号溢出 |

### Y86-64 指令类型

Y86-64 的指令大致分为以下几类：

1. **算术/逻辑指令**：`addq`, `subq`, `xorq`, `andq` 等，操作两个寄存器，设置条件码
2. **数据传送指令**：
   - `irmovq Imm, Reg`：立即数到寄存器
   - `rrmovq Reg, Reg`：寄存器到寄存器
   - `mrmovq Offset(Reg), Reg`：内存到寄存器
   - `rmmovq Reg, Offset(Reg)`：寄存器到内存
3. **控制流指令**：`jmp`, `je`, `jne`, `jl`, `jle`, `jg`, `jge` 等
4. **函数调用与返回**：`call`, `ret`
5. **系统指令**：`halt` 停止执行
6. **无操作指令**：`nop`，不执行任何操作

## 4.3 指令执行阶段详解

### 取指（Fetch）

- 根据 PC 从内存读取指令
- 解析指令的两部分：
  - `icode`：指令代码（4 位）
  - `ifun`：指令功能（4 位）
- 如果指令含有寄存器操作数，读取寄存器指示符 `rA` 和 `rB`
- 如果指令含有立即数，读取 64 位常数 `valC`
- 计算下一条指令的地址 `valP = PC + 指令长度`

### 译码（Decode）

从寄存器文件读取 0 到 2 个操作数：

- `valA`：通常是寄存器 `rA` 的值
- `valB`：通常是寄存器 `rB` 的值
- 某些指令（如 `pushq`, `popq`, `call`, `ret`）会隐式读取 `%rsp`

### 执行（Execute）

- 执行 ALU 运算，结果记为 `valE`
- 计算内存访问地址（如果指令涉及内存操作）
- 设置条件码（对算术/逻辑指令）
- 判断条件跳转是否应该发生（依据条件码和 `ifun`）

### 访存（Memory）

- 如果指令涉及读取，从指定地址读取数据，存到 `valM`
- 如果指令涉及写入，向指定地址写数据
- `ret` 指令在此阶段从栈顶地址读出返回地址（`valM = M[R[%rsp]]`）

### 写回（Write Back）

将计算结果写回寄存器文件（最多写两个目标寄存器，例如 `popq` 同时更新目标寄存器和 `%rsp`）。

### 更新 PC（Update PC）

- 对于正常指令：`PC = valP`
- 对于条件/无条件跳转（若跳转）：`PC = valC`（目标地址）
- 对于 `ret` 指令：`PC = valM`（从栈上读出的返回地址）

# 第二部分 流水线处理

## 4.4 流水线原理

流水线的核心思想是**任务并行**：多条指令在不同阶段同时执行，就像工业装配线一样。

### 吞吐量与延迟

流水线提升的是**吞吐量**（throughput）——单位时间内完成的指令数。但单条指令从取指到写回的**延迟**（latency）实际上因为流水线寄存器的引入而略有增加。理解这一区分是认识"流水线加速本质"的前提，也是"流水线不能无限加深"的原因之一（见 [§4.5](#45-流水线的效率挑战)）。

### 顺序执行（SEQ）

```
指令1：取指 -> 译码 -> 执行 -> 访存 -> 写回 -> 更新PC
       |-------- 每条指令需要完整走完所有阶段 -------|
                                                     ↓
指令2：                                             取指 -> ...
```

### 流水线执行（PIPE, pipeline）

在 PIPE 处理器中，Update PC 的逻辑被折叠进 Fetch 阶段，因此流水线只有 **F、D、E、M、W** 五个阶段：

| 时钟周期 |  1  |  2  |  3  |  4  |  5  |  6  |  7  |  8  |
| :------: | :-: | :-: | :-: | :-: | :-: | :-: | :-: | :-: |
|  指令1   |  F  |  D  |  E  |  M  |  W  |     |     |     |
|  指令2   |     |  F  |  D  |  E  |  M  |  W  |     |     |
|  指令3   |     |     |  F  |  D  |  E  |  M  |  W  |     |
|  指令4   |     |     |     |  F  |  D  |  E  |  M  |  W  |

理想情况下，流水线可以在每个时钟周期完成一条新指令，显著提高吞吐量。

### 流水线寄存器

为了在不同阶段之间传递信息，相邻阶段之间插入**流水线寄存器**，在时钟上升沿保存本阶段的输出供下一阶段使用。PIPE 共有五个流水线寄存器，按其所处位置命名：

| 寄存器 | 位置                     | 保存内容                                                          |
| ------ | ------------------------ | ----------------------------------------------------------------- |
| F      | Fetch 阶段内部           | `predPC`：预测的下一条指令地址（仅此一个值）                      |
| D      | Fetch → Decode 之间      | `icode`, `ifun`, `rA`, `rB`, `valC`, `valP`, `stat`               |
| E      | Decode → Execute 之间    | `valA`, `valB`, `valC`, `dstE`, `dstM`, `srcA`, `srcB`, `stat` 等 |
| M      | Execute → Memory 之间    | `valE`, `Cnd`（跳转条件），`valA`, `dstE`, `dstM`, `stat` 等      |
| W      | Memory → Write Back 之间 | `valE`, `valM`, `dstE`, `dstM`, `stat`                            |

:::note
寄存器 F 只存一个预测 PC，真正的指令字段（icode 等）保存在寄存器 D 中；Write Back 之后没有流水线寄存器。
:::

## 4.5 流水线的效率挑战

### 4.5.1 不一致的阶段延迟

在理想流水线中，每个阶段耗时相同，时钟周期等于阶段延迟的最大值。但实际上：

- 取指、译码、写回的逻辑相对简单（延迟低）
- 执行和访存的逻辑复杂（延迟高）

为了对齐，每个阶段必须等待最慢的阶段，时钟周期被迫设置为最大阶段延迟。这就浪费了快速阶段的潜力，即使没有[冒险](#第三部分-流水线冒险与解决方案)，流水线的效率也达不到 100%。

### 4.5.2 流水线深度的权衡

增加流水线深度（阶段数更多）可以降低每个阶段的延迟，提升时钟频率上限，但代价是：

- 流水线寄存器增加，每个寄存器的读写本身引入额外延迟，单条指令的端到端延迟上升
- 冒险导致的停顿惩罚更重（指令间距更远，转发路径更长）
- 分支预测错误时需要清空更多阶段，代价更大

这就是**吞吐量与延迟的根本矛盾**：加深流水线能提高吞吐量上限，但每条指令的完成时间（延迟）随之增大，预测失败和停顿的代价也同步上升。实际处理器通常采用 10～15 级流水线作为折中。

# 第三部分 流水线冒险与解决方案

流水线的最大挑战是**冒险（hazard）**：后续指令需要前面指令的结果，但前面指令还未完成。冒险分为两类。

## 4.6 数据冒险

### 什么是数据冒险

考虑以下指令序列：

```asm
addq  %rax, %rbx       # 结果写入 %rbx
mrmovq 0(%rbx), %rdx   # 需要读 %rbx
addq  %rdx, %r8        # 需要读 %rdx
```

- 第 2 条指令需要第 1 条的执行结果（`%rbx`）
- 第 3 条指令需要第 2 条的执行结果（`%rdx`）

这就是**数据相关（data dependency）**，会导致**数据冒险（data hazard）**。

### 冒险的类型

在标准 5 级流水线中，数据冒险分为三类：

| 冒险类型                | 前后指令         | 说明                               |
| ----------------------- | ---------------- | ---------------------------------- |
| Read-After-Write (RAW)  | 后指令读前指令写 | 最常见的数据依赖                   |
| Write-After-Read (WAR)  | 后指令写前指令读 | 流水线中通常不会发生（写回在后期） |
| Write-After-Write (WAW) | 后指令写前指令写 | 相同寄存器被多次写                 |

:::note
RAW 冒险是最常见的，也是最需要解决的冒险类型。WAR 和 WAW 冒险在简单 5 级流水线中不会发生，因为写回阶段在后面，读操作在前面。
:::

## 4.7 控制冒险

### 什么是控制冒险

分支和返回指令的目标地址不确定，导致后续指令不知道应该取哪一条。

条件分支：

```asm
je   .L1               # 条件分支，跳转结果在执行（E）阶段才确定
addq %rax, %rbx        # 这条指令该执行吗？
```

`ret` 指令更复杂：

```asm
ret                    # 返回地址在访存（M）阶段才从栈上读出
```

这就是为什么简单 5 级流水线中 `ret` 需要 3 个气泡：当 ret 在执行（E）阶段时，已经错误取了 3 条后续指令；而直到 ret 进入访存（M）阶段，返回地址才从 `M[R[%rsp]]` 中读出，此时才能更正 PC。

## 4.8 冒险的解决方法

### 4.8.1 暂停（Stalling）

遇到冒险时，在相关指令前插入**气泡（bubble）**——一个无操作的占位符，暂停流水线直到所需的值已经就绪。

以 RAW 冒险为例（不使用转发时，最多需要 2 个气泡）：

| 时钟周期 |  1  |  2  |  3  |   4    |   5    |  6  |  7  |  8  |
| :------: | :-: | :-: | :-: | :----: | :----: | :-: | :-: | :-: |
|  指令1   |  F  |  D  |  E  |   M    |   W    |     |     |     |
|  指令2   |     |  F  |  D  | bubble | bubble |  E  |  M  |  W  |

优点：简单易实现。缺点：性能下降明显，尤其是对频繁发生的冒险。

### 4.8.2 数据转发（Data Forwarding）

与其等待指令写回寄存器文件，不如**直接从流水线寄存器把结果传给后续指令的执行阶段**，无需经过寄存器文件的读写流程。

转发的目标是当前处于**执行阶段（E）的指令的操作数输入**，绕过寄存器文件的读取延迟。转发来源共有五条路径：

| 来源             | 值                             | 可消除的冒险距离    |
| ---------------- | ------------------------------ | ------------------- |
| E/M 流水线寄存器 | `e_valE`（ALU 刚计算出的结果） | 紧邻（距离 1）      |
| M/W 流水线寄存器 | `m_valE`（执行结果传递中）     | 距离 2              |
| M/W 流水线寄存器 | `m_valM`（内存读取结果）       | 距离 2（load 指令） |
| W 阶段           | `W_valE`                       | 距离 3              |
| W 阶段           | `W_valM`                       | 距离 3（load 指令） |

转发可以消除大部分 RAW 冒险，但**无法完全消除**（见 [§4.8.3](#483-加载使用冒险load-use-hazard)）。

### 4.8.3 加载/使用冒险（Load-Use Hazard）

一个重要的例外：

```asm
mrmovq 0(%rax), %rbx   # 从内存读数据到 %rbx（M 阶段才得到结果）
addq   %rbx, %rcx      # 紧接着使用 %rbx
```

内存读取结果 `valM` 在访存（M）阶段才产生，而紧跟其后的使用指令此时已经进入执行（E）阶段，需要操作数。时序上无法直接转发——当 load 在 E 阶段时，使用指令在 D 阶段，此时 load 的结果根本还没算出来。

解决方案是**停顿 1 个周期 + 转发**：

1. 检测到 load-use 时，插入 1 个气泡，使用指令暂停在 D 阶段
2. 下一个周期，load 进入 M 阶段产生 `m_valM`
3. 通过转发路径将 `m_valM` 送到使用指令的执行阶段输入

硬件中的检测器（load interlock）负责识别这种情况并自动插入气泡。编译器也可以通过在 load 和使用之间插入无关指令来消除这 1 个周期的停顿（指令调度）。

### 4.8.4 分支预测（Branch Prediction）

对于条件分支，不能等到执行阶段才知道结果，否则会白白浪费流水线。PIPE 处理器采用的默认策略是**预测不跳转（predict not taken）**：

- 无条件跳转（`jmp`）：总是预测跳转（因为一定会跳）
- 条件分支（`je`, `jl` 等）：默认预测不跳转，继续顺序取指

具体流程：

1. 取出分支指令，预测不跳转，继续顺序取后续指令
2. 执行阶段确认真实结果
3. 若预测正确：继续执行，无代价
4. 若预测错误：**取消（squash）** 已错误流入的指令——将 D 和 E 流水线寄存器中的内容替换为 nop 气泡，同时将 PC 修正为正确的跳转目标地址。此操作与"停顿"使用不同机制，停顿只是暂停推进，取消是主动清除错误内容。

`ret` 指令由于返回地址存在栈上，无法提前预测，通常需要 **3 个气泡**的停顿（在简单 5 级流水线中）。更复杂的处理器会用**返回地址栈**（return address stack）来预测 `ret` 的目标，大幅降低停顿代价。

# 第四部分 性能分析

## 4.9 CPI 与性能指标

### CPI（Cycles Per Instruction）

**CPI** 表示平均每条指令需要多少个时钟周期完成。

设某个时间段内：

- 执行的指令数：$C_i$
- 产生的气泡数：$C_b$
- 总时钟周期数：$C_i + C_b$

则：

$$
\text{CPI} = \frac{C_i + C_b}{C_i} = 1 + \frac{C_b}{C_i}
$$

**理想情况**：CPI = 1（每个周期 1 条指令）  
**实际情况**：CPI > 1（因为各种冒险导致停顿）

### 气泡的来源

气泡主要来自三方面：

1. **加载/使用冒险**：约 20% 的指令是 load，其中约 25% 紧接着使用，引起 1 个气泡
2. **分支预测错误**：错预测时需要取消 2 条已流入的指令（2 个气泡）
3. **`ret` 指令**：每个 `ret` 需要 3 个气泡

对于典型程序的 CPI 估算：

| 来源                       | 计算                        | 贡献       |
| -------------------------- | --------------------------- | ---------- |
| load-use 冒险              | $0.20 \times 0.25 \times 1$ | 0.05       |
| 分支预测错误（错误率 10%） | $0.20 \times 0.10 \times 2$ | 0.04       |
| `ret` 指令                 | $0.05 \times 3$             | 0.15       |
| 合计                       |                             | **≈ 1.24** |

### 吞吐量与延迟的区分

CPI 衡量的是**吞吐量**维度的性能。需要注意，流水线提升的是吞吐量，单条指令的端到端**延迟**实际上因为流水线寄存器的引入而略有增加（每个寄存器都有读写开销）。两个维度的区分对理解性能优化方向至关重要：

- 要提高程序总执行速度（吞吐量）：降低 CPI、提高时钟频率
- 要降低单条指令的响应时间（延迟）：减少流水线级数或使用乱序执行

### 性能优化方向

为了降低 CPI（提高吞吐量）：

1. **改进分支预测准确率**：减少预测错误的取消代价
2. **增加转发路径**：减少 load-use 冒险的停顿
3. **使用返回地址栈预测 `ret`**：避免 3 个气泡的停顿
4. **指令调度**：编译器重排指令，在 load 和使用之间插入无关指令，消除 load-use 停顿
